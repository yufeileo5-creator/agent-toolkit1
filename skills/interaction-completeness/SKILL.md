# 交互完整性审查 (Interaction Completeness Review)

## 核心原则

**功能写完 ≠ 交付完成。** 每个用户可感知的功能都必须通过以下检查清单，才能被视为"完成"。

## 何时触发

- 新增面向用户的 UI 功能或组件后
- 修改现有交互行为后
- 用户反馈"能用但不好用"时
- Code Review 前的自查阶段

## 强制检查清单

完成功能编码后，**逐项对照以下清单**，遗漏项必须补全。

---

### 1. 五态完整性 (Five States)

每个数据驱动的 UI 区域必须覆盖全部五种状态：

| 状态 | 检查点 | 常见遗漏 |
|------|--------|----------|
| **空状态 (Empty)** | 无数据时显示引导性提示和操作入口 | 只显示空白或 "No data" |
| **加载状态 (Loading)** | 使用骨架屏或 Spinner，尺寸匹配实际内容 | 无任何加载反馈，页面"卡住" |
| **错误状态 (Error)** | 显示具体错误原因 + 可操作的恢复按钮 | 只 console.error，用户无感知 |
| **成功状态 (Success)** | 操作结果的视觉反馈（Toast/动画/状态变化） | 操作完成后毫无反应 |
| **禁用状态 (Disabled)** | 不可操作时视觉灰化 + Tooltip 解释原因 | 按钮可点但无反应 |

---

### 2. 边界条件 (Edge Cases)

| 检查点 | 问自己 |
|--------|--------|
| **超长内容** | 文本溢出时是 truncate/clamp 还是撑破布局？ |
| **极端数值** | 0、负数、超大数字是否正常显示？ |
| **快速连续操作** | 高频点击按钮是否会重复提交？（debounce/throttle） |
| **并发冲突** | 多处同时修改同一数据，谁先谁后？ |
| **空输入/特殊字符** | 空字符串、纯空格、XSS 字符是否被处理？ |
| **网络异常** | 断网/超时/服务端 500 时，用户看到什么？ |

---

### 3. 键盘与焦点 (Keyboard & Focus)

| 检查点 | 要求 |
|--------|------|
| **Tab 顺序** | 所有交互元素可被 Tab 到达，顺序符合视觉流 |
| **Enter/Space** | 按钮和链接可通过键盘激活 |
| **Escape** | 弹窗/下拉/浮层可通过 Escape 关闭 |
| **焦点陷阱** | Modal 打开时焦点锁定在内部；关闭后焦点回到触发元素 |
| **焦点可见** | 键盘聚焦时必须有清晰的 focus ring（不得被 `outline: none` 抹掉） |
| **快捷键** | 若有自定义快捷键，是否覆盖了浏览器原生行为？ |

---

### 4. 拖放与画布交互 (Drag & Canvas，如适用)

| 检查点 | 要求 |
|--------|------|
| **拖放边界** | 拖拽元素是否被限制在合理区域内？ |
| **缩放极限** | 缩放比例是否有 min/max 限制？ |
| **碰撞检测** | 拖放到重叠位置时的行为是否明确？ |
| **拖放反馈** | 拖拽过程中是否有视觉指示（Ghost/占位符）？ |
| **多选操作** | 批量选中后的操作是否一致？ |
| **撤销/重做** | 拖放、删除等破坏性操作是否可撤销？ |
| **坐标精度** | 缩放/平移后坐标计算是否准确？ |

---

### 5. 响应式与适配 (Responsive)

| 检查点 | 要求 |
|--------|------|
| **移动端触摸** | 触摸区域 ≥ 44×44px |
| **断点行为** | 组件在 sm/md/lg 断点下的布局是否合理？ |
| **滚动行为** | 内部滚动区域是否阻止了外部页面滚动？ |
| **虚拟键盘** | 输入时虚拟键盘弹出是否遮挡内容？ |

---

### 6. 过渡与反馈 (Transitions & Feedback)

| 检查点 | 要求 |
|--------|------|
| **操作反馈** | 按钮点击后是否有即时视觉响应（`active` 态、loading spinner）？ |
| **状态切换** | 从一个视图/模式切换到另一个时，是否有平滑过渡？ |
| **进度指示** | 长时间操作是否有进度条或步骤指示器？ |
| **确认提示** | 删除/覆盖等不可逆操作是否弹出确认框？ |
| **未保存提醒** | 离开有未保存数据的页面时是否拦截提醒？ |

---

### 7. 文案与提示 (Copy & Messaging)

| 检查点 | 要求 |
|--------|------|
| **错误信息** | 具体、可操作（不是 "Something went wrong"） |
| **空状态引导** | 告诉用户如何开始，而不只是说"暂无数据" |
| **Tooltip** | 图标按钮必须有 Tooltip 说明功能 |
| **Placeholder** | 输入框 Placeholder 应展示格式示例（如 "2026-03-09"） |
| **操作确认** | 成功/失败的 Toast 信息简洁明确 |

## 使用方法

### 在编码阶段（事前）

编写新功能时，在脑中过一遍清单中与该功能相关的部分，确保代码中覆盖了对应的状态和分支。

### 在交付阶段（事后）

功能完成后，逐节逐项对照检查：

```
✅ 通过  — 已实现且正常工作
⚠️ 部分  — 已实现但需要改进
❌ 缺失  — 完全没有处理
🚫 不适用 — 该功能不涉及此项
```

将所有 ❌ 和 ⚠️ 项修复后，才能宣布功能完成。

### 自动化集成

在 Code Review 或 `/plan` 工作流中，将此检查清单作为交付门控的一部分。
功能 PR 描述中应包含一段"交互完整性自查"确认。

## 常见反模式

| 反模式 | 正确做法 |
|--------|----------|
| 只处理 happy path | 每个异步操作都写 `try/catch`，每个判断都考虑 else |
| 错误吞掉不显示 | `catch` 中必须有用户可见的反馈（Toast/inline error） |
| 按钮无防抖 | 提交按钮至少用 `disabled` + loading 状态防止重复点击 |
| Modal 无 ESC 关闭 | 用 Dialog 组件的标准关闭行为 |
| 删除无确认 | 所有不可逆操作必须二次确认 |
| 拖放无边界 | 拖拽必须设置 bounds，缩放必须设置 min/max |
| 表单无校验反馈 | 校验失败时高亮对应字段 + 显示错误信息 |

## 与其他 Skills 的协作

- **baseline-ui**: 负责基础 CSS/动画约束 → 本 skill 在更高层面检查交互流程完整性
- **taste-skill**: 负责视觉审美 → 本 skill 确保"好看"之外还"好用"
- **verification-before-completion**: 负责验证命令执行 → 本 skill 补充交互层面自查
- **webapp-testing**: 负责 E2E 测试 → 本 skill 在写测试前先确保交互逻辑不遗漏
