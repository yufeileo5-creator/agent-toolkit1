# Agent Eval — Agent 质量评估

在任务完成后系统化评估 Agent 表现，识别 Harness 薄弱环节，驱动持续改进。

## 触发时机

- 重大功能开发完成后
- 对话交接（`/handoff`）前
- 用户主动触发：「评估一下表现」、「agent eval」
- 连续出现 2+ 次 pua 触发后

## 评估维度

### 维度 1：指令遵循率 (Instruction Compliance Rate)

检查全局规则和 AGENTS.md 中的关键指令是否被遵守：

| 检查项 | 检查方法 |
|--------|---------|
| 中文输出 | 对话中的分析、说明、代码注释是否全部中文 |
| TDD 铁律 | 新功能是否先写了测试再写实现 |
| 禁用 any | 新代码中是否出现 any 类型 |
| 占位符禁令 | 代码块中是否出现 `// ... existing code` |
| 版本保护 | 修改核心文档时是否先归档旧版本 |
| 分层约束 | 是否触发了 DSP 或架构检查 |

**计算**: 遵守条数 / 适用条数 × 100%

### 维度 2：首次通过率 (First-Pass Success Rate)

衡量 Agent 在不借助 pua 的情况下一次完成任务的能力：

| 指标 | 计算方法 |
|------|---------|
| 构建首次通过 | `tsc --noEmit` 第一次运行是否零错误 |
| 测试首次通过 | `npx vitest run` 第一次运行是否全部通过 |
| 功能首次验证 | 用户是否对第一次交付表示满意（无返工） |
| pua 触发次数 | 本次任务中 pua 技能被触发的次数 |

**计算**: 首次通过的子任务数 / 总子任务数 × 100%

### 维度 3：架构合规率 (Architecture Compliance Rate)

| 检查项 | 检查方法 |
|--------|---------|
| 分层依赖 | 运行 GR-A001 ~ GR-A004 检查 |
| 命令总线协议 | 运行 GR-A005 ~ GR-A006 检查 |
| 钩子预注册 | 运行 GR-A007 检查 |
| DSP 同步 | 运行 GR-A008 ~ GR-A009 检查 |

**计算**: 通过的准则数 / 适用的准则数 × 100%

### 维度 4：文档同步率 (Documentation Sync Rate)

| 检查项 | 检查方法 |
|--------|---------|
| feature-map 更新 | GR-D001 / GR-D002 |
| CHANGELOG 追加 | GR-D003 |
| architecture.md 更新 | GR-D004 |
| AGENTS.md 更新 | GR-D005 |
| DSP 图谱同步 | GR-D006 |
| ADR 创建 | GR-D008（如适用） |

**计算**: 已同步的文档数 / 需同步的文档数 × 100%

## 输出格式

```markdown
# 📊 Agent 质量评估报告

> 评估时间: YYYY-MM-DD HH:mm
> 评估范围: [任务名称/功能描述]

## 综合评分

| 维度 | 得分 | 等级 |
|------|------|------|
| 指令遵循率 | XX% | 🟢/🟡/🔴 |
| 首次通过率 | XX% | 🟢/🟡/🔴 |
| 架构合规率 | XX% | 🟢/🟡/🔴 |
| 文档同步率 | XX% | 🟢/🟡/🔴 |
| **综合得分** | **XX%** | **🟢/🟡/🔴** |

### 评分标准
- 🟢 ≥ 85%: 优秀
- 🟡 60~84%: 合格（需改进）
- 🔴 < 60%: 不合格（需要复盘）

## Harness 薄弱环节

> 以下是本次暴露出的 Harness 改进点

1. [薄弱环节描述 + 建议补强措施]

## 改进建议

> 如果识别出反复出现的问题模式，建议：

- 新增 Golden Rule → `.golden-rules/`
- 更新 AGENTS.md 上下文
- 新增 AGENT_HINT 模板 → `.golden-rules/error-hints.golden.md`
```

## 评分等级行为

| 综合得分 | 后续行为 |
|---------|---------|
| 🟢 ≥ 85% | 正常继续，记录优秀模式 |
| 🟡 60~84% | 生成改进建议，建议更新 AGENTS.md |
| 🔴 < 60% | 强制触发 Harness 自修复：分析失败模式 → 更新准则 → 沉淀到仓库 |

## 最佳实践

- ✅ 评估应基于客观检查结果，而非主观判断
- ✅ 评估报告应保存到对话 artifact 中，方便追踪趋势
- ✅ 低分维度应转化为具体的 Harness 改进行动
- ❌ 不要在评估中修改代码（评估和修复应分离）
- ❌ 不要忽略黄色等级（60~84% 是退化信号）

## 搭配使用

- `harness-gc` — 针对架构合规和文档同步的深度扫描
- `pua` — 评估结果可作为 pua 压力等级的参考
- `verification-before-completion` — 在评估前确保验证已完成
