# 回归守卫 (Regression Guard)

## 概述

实现新功能后破坏旧功能，是回归缺陷（Regression Bug）的典型表现。
根因是：**修改代码时不知道影响了谁，也没有验证旧功能是否完好。**

本技能通过 **三层防护** 解决这个问题：

| 层级 | 手段 | 可靠性 | 说明 |
|------|------|--------|------|
| 📊 代码级 | `git diff` + 导入链分析 | ⭐⭐⭐⭐⭐ | 最实时，看实际改了什么 |
| 📄 图谱级 | DSP 依赖关系查询 | ⭐⭐⭐ | 辅助参考，可能过时 |
| 🧪 测试级 | 跑测试 + 构建 | ⭐⭐⭐⭐ | 只要覆盖到就可靠 |

**核心原则：** 证据优先于假设。不跑测试就不能说"没问题"。

## 何时触发

- 修改任何已有代码文件时
- 添加新功能且涉及现有模块时
- 重构、优化、调整已有逻辑时
- 修复 Bug 时（修一个 Bug 可能引入新 Bug）
- 删除或重命名导出的函数/组件/接口时

**不适用场景：**
- 纯新建文件，不涉及任何已有模块
- 仅修改文档/注释，不改逻辑

## 流程

### Step 1: 变更扫描

**分析实际改了什么，这是最可靠的起点。**

1. 执行 `git diff --name-only` 获取变更文件列表
2. 执行 `git diff --stat` 了解变更规模
3. 对每个变更文件，分析具体改动：
   - 修改了哪些函数/组件？
   - 修改了哪些导出接口？
   - 修改了哪些类型定义？
   - 是否删除/重命名了公共 API？

4. 将变更分类：

| 变更类型 | 风险等级 | 说明 |
|---------|---------|------|
| 删除/重命名导出 | 🔴 高危 | 所有消费者都会受影响 |
| 修改函数签名 | 🔴 高危 | 调用方可能传参错误 |
| 修改返回值结构 | 🟡 中危 | 下游解构可能失败 |
| 修改内部逻辑 | 🟡 中危 | 行为变化可能影响依赖方 |
| 修改样式/布局 | 🟢 低危 | 影响范围有限 |
| 新增导出 | 🟢 低危 | 不影响现有代码 |

> ⚠️ **如果存在 🔴 高危变更，必须立刻标注 `⚠️ BREAKING CHANGE` 并通知用户确认。**

### Step 2: 影响推导

**找出哪些模块会被你的修改影响。**

1. **导入链分析（主要手段）**：
   - 对每个变更文件，搜索项目中有谁 `import` 了它
   - 递归追踪：A 改了 → B 导入了 A → C 导入了 B → C 也受影响
   - 使用 `grep -r "from.*变更文件名"` 或 IDE 的引用查找

2. **DSP 图谱查询（辅助手段）**：
   - 如果项目有 `.dsp/` 目录，查询变更实体的依赖关系
   - **注意：** 图谱可能不是最新的，以代码实际导入为准
   - 如果图谱与实际代码不一致，记录下来，在 Step 5 同步

3. **输出影响清单**：

```
## 影响分析报告

### 直接变更
- `文件A.ts` — 修改了 `函数X` 的返回值

### 受影响模块（一级）
- `文件B.ts` — 导入了 `函数X`
- `文件C.tsx` — 使用了 `函数X` 的返回值

### 受影响模块（二级）
- `页面D.tsx` — 渲染了 `文件C` 组件
```

### Step 3: 回归检查清单

**逐项确认，不遗漏。**

参照 📄 **[references/impact-checklist.md](references/impact-checklist.md)** 中的完整清单，针对每个受影响模块检查：

| # | 检查项 | 状态 |
|---|--------|------|
| 1 | 受影响模块的导入语句是否仍然有效？ | ✅/❌ |
| 2 | 受影响模块使用的 API 签名是否匹配？ | ✅/❌ |
| 3 | 受影响模块的类型定义是否兼容？ | ✅/❌ |
| 4 | 受影响模块的行为预期是否不变？ | ✅/❌ |
| 5 | 是否有对应的测试覆盖该模块？ | ✅/❌ |

> 🛡️ 如果第 5 项为 ❌（没有测试覆盖），**必须先为该模块补写测试**，再继续。
> 使用 `test-driven-development` 技能编写测试。

### Step 4: 测试验证

**用证据证明没有回归。**

1. **运行全量测试**：
   ```bash
   npm run test:run    # 或项目对应的测试命令
   ```
   - 确认所有测试通过
   - 如有失败，**立刻停止**，回到 Step 1 重新分析

2. **运行构建**：
   ```bash
   npm run build       # 或项目对应的构建命令
   ```
   - 确认 TypeScript 编译无错误
   - 确认构建产物正常

3. **类型检查**：
   ```bash
   npx tsc --noEmit    # TypeScript 类型检查
   ```

4. **如果没有自动化测试**：
   - 手动逐一测试影响清单中的每个模块
   - 记录测试步骤和结果
   - **为关键路径补写自动化测试**

> 📋 使用 `verification-before-completion` 技能确保验证是真实运行的，不是"应该没问题"。

### Step 5: 图谱同步

**保持架构记忆是最新的。**

如果项目有 `.dsp/` 目录，且本次变更涉及：
- 新建了模块/文件
- 修改了导出接口
- 增删了依赖关系

则必须使用 `data-structure-protocol` 技能同步更新 DSP 图谱。

如果 Step 2 中发现图谱与实际代码不一致，也要在此步骤修复。

### Step 6: 完成报告

向用户输出以下格式的报告：

```markdown
## 回归守卫报告

### 变更概览
| 文件 | 变更类型 | 风险等级 |
|------|---------|---------|
| {文件} | {类型} | {等级} |

### 影响分析
- 直接影响：{N} 个模块
- 间接影响：{N} 个模块
- 破坏性变更：{有/无}

### 测试验证
- 测试：{全 N 项通过 / N 项失败}
- 构建：{成功 / 失败}
- 类型检查：{通过 / N 个错误}

### 图谱同步
- DSP 更新：{已更新 / 无需更新 / 项目无 DSP}
```

## 红旗信号 — 立刻停下

如果你发现自己在想：

- "这个改动很小，应该不会影响其他地方" → **做影响分析**
- "测试太慢了，先跳过" → **必须跑测试**
- "这个模块没有测试，算了" → **先补测试**
- "DSP 图谱太旧了，不看了" → **用导入链分析替代**
- "构建通过就行了" → **构建通过 ≠ 功能正常**

**以上任何一条出现，说明你正在走向回归缺陷。STOP。回到 Step 1。**

## 反模式

| ❌ 错误做法 | ✅ 正确做法 |
|------------|------------|
| 只看改动的文件，不看影响链 | 追踪完整导入链到末端消费者 |
| 只跑修改文件的测试 | 跑全量测试，尤其是受影响模块的测试 |
| 依赖过时的文档判断影响 | 以 `git diff` + 实际导入为准 |
| 改完就说"完成了" | 跑测试、跑构建、出报告再说完成 |
| 补丁式修复，不查根因 | 用 `systematic-debugging` 找根因 |

## 关联技能

| 技能 | 联动场景 |
|------|---------|
| `test-driven-development` | Step 3 中发现缺失测试时，触发 TDD 补写 |
| `verification-before-completion` | Step 4 中验证测试和构建结果 |
| `systematic-debugging` | 测试失败时，触发四阶段诊断 |
| `data-structure-protocol` | Step 5 中同步 DSP 图谱 |
| `code-review` | 变更较大时，触发六大特工代码审查 |
