# 僵尸代码清理工作流 (Dead Code Sweeper)

处理僵尸代码是一项高风险操作。为了防止误删动态引用的代码、保护版本回退的清晰度，**绝对禁止在开发新功能的途中"顺手"删除看似无用的代码。** 清理必须作为独立的任务执行。

## 开发中发现僵尸代码怎么办？

如果在开发新功能时，发现了疑似僵尸代码：
1. **绝不动手删除**。
2. 在当前的 commit message 或 PR 描述中用 `// TODO(dead-code): 疑似未使用` 标记。
3. 继续完成当前功能开发。
4. 功能完成并提交后，再单独发起清理任务。

---

当用户要求"扫描僵尸代码"或"清理无用代码"时，严格按照以下四个阶段执行：

## 阶段 1：全局扫描与侦查 (Scan & Detect)

在此阶段，你的目标是生成一份《僵尸代码嫌疑清单》，**绝对不要修改任何代码**。

### 1. 使用 knip 进行静态依赖分析
项目已安装 `knip`，直接运行：
```bash
npm run find-deadcode
```
它会报告：
- 未使用的文件 (Unused files)
- 未使用的导出 (Unused exports)
- 未使用的依赖包 (Unused dependencies)
- 未列入但被引用的依赖 (Unlisted dependencies)

### 2. DSP 孤岛检测
查询 `.dsp/` 目录中的关系图谱，找出 `incoming edges`（入度）为 0 的节点（即没有任何模块依赖它），且它不是项目的入口文件（如 `App.tsx`、路由文件等）。

### 3. 全局文本搜索防误杀
对于上两步找出的嫌疑代码，必须使用全局文本搜索（忽略 `node_modules` 和 `dist`）进行最后确认：
- 防止动态导入 (`import()`) 造成的漏判。
- 防止通过字符串拼接调用的组件/类名。

---

## 阶段 2：制定清理计划 (Plan & Approve)

将《僵尸代码嫌疑清单》整理到 `implementation_plan.md` 中，并严格走 `/plan` 审批流程。

清单格式必须包括：
*   **高确信度（可安全删除）**：完全孤立的文件或未被使用的死函数。
*   **疑似/需确认（动态风险）**：可能通过反射、动态路由或字符串拼接被引用的代码。
*   **陈旧配置/依赖**：`package.json` 中多余的依赖。

**向用户提交此计划，并设置 `BlockedOnUser: true`，等待用户确认哪些可以删。**

---

## 阶段 3：原子化切除与重构 (Atomic Refactor)

获得用户批准后，开始执行切除手术：

### 1. 从末端开始
删除没有依赖其他嫌疑代码的"叶子节点"，自下而上进行清理，防止中途报错。

### 2. 隔离的 Commit
清理僵尸代码的 Git 提交**必须独立**。如果是多个无关的模块，建议分批提交：
- `refactor: 移除废弃的旧版卡片组件`
- `chore: 移除未使用的 lodash 依赖`
*绝对不要和新功能混在一个 commit 里！*

---

## 阶段 4：严格验证与同步 (Verify & Sync)

### 1. 绿灯验证 (Green Light)
删除完成后，必须在终端运行：
- 类型检查（如 `npx tsc --noEmit` 或 `npm run lint`）
- 单元测试/E2E 测试（如果有）
如果在任何一步报错，说明**误杀了非僵尸代码**，必须立刻回退 (`git restore` 或还原修改) 并排查。

### 2. 架构图谱同步
在确认代码正常运行后，使用 `dsp-cli` 将被删除的模块、依赖从 `.dsp/` 的架构图谱中移除，保持地图与现实一致。

### 3. 反馈记录
在 `walkthrough.md` 中记录本次清理掉了哪些模块，释放了多少空间或减少了哪些维护负担。
