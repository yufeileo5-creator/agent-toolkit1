# 可行性与硬边界审查 (Feasibility Check)

当你收到用户提出的新功能需求，特别是涉及底层改动、浏览器机制、性能极高要求或未记录的 API 交互时，**在编写任何实现代码之前，必须强制触发此 Skill。**

## 豁免条件（以下情况无需触发）

- 纯 CSS 样式微调（颜色、间距、字号等）
- 文案/文本内容修改
- 已在项目中有成熟先例的同类功能扩展（如：已有 TextBlock，再加一个类似的 QuoteBlock）
- Bug 修复（已知功能的回归修复，不涉及新架构设计）

**如果不确定是否豁免，默认触发此 Skill。宁可多检查一次，不可漏判一次。**

## 核心原则："Fail Fast" (尽早失败)

你的首要任务是**保护用户的时间和项目架构**。
如果你发现一个需求：
1. **根本做不到**（如：浏览器安全沙箱限制、DOM 无法突破的物理限界）。
2. **代价极高且收益极低**（如：需要重写整个状态管理核心只为了一个微小的动画）。
3. **存在完美现成的替代方案**，没必要造轮子。

**你必须立刻停止！不要写代码去"试试看"！直接告诉用户真相！**

## 执行流程

在回复用户时，严格按照以下三步输出你的评估结果：

### 阶段 1：边界扫描 (扫描引擎限制)
分析此功能是否触碰了以下"死胡同"：

**浏览器与前端层：**
- **浏览器安全限制** (CORS, 跨域 iframe, 读取本地非授权文件, 绕过 HTTPS 限制等)
- **前端框架限制** (在 Server Component 中使用 Hooks, React 状态更新死循环, 强制同步读取异步流等)
- **性能红线** (主线程阻塞超过 50ms, 频繁引起重排/重绘的巨量 DOM 节点操作等)
- **现有第三方库死角** (当前引入的库如 Mammoth.js、html2canvas 是否明确不支持此操作)

**AI / LLM 层：**
- **模型能力边界** (LLM 无法做到像素级精确控制、无法保证 100% 确定性输出、无法实时感知用户屏幕)
- **API 限制** (Gemini API 的 token 上限、速率限制、不支持的输入格式如音视频流等)
- **生成质量的不确定性** (AI 生成的布局/文本天然存在随机性，不能承诺"每次都完美一致")

**桌面端 / Node.js 层（如涉及 ai-bridge-desktop 或 ai-bridge-cli）：**
- **Electron 主进程与渲染进程隔离** (渲染进程无法直接访问文件系统、主进程无法直接操作 DOM)
- **Node.js 原生模块兼容性** (某些 C++ 原生模块在不同平台上的编译问题)
- **跨平台差异** (Windows/macOS/Linux 上的 Shell 命令、路径分隔符、权限模型差异)

### 阶段 2：评级判定
根据扫描结果，给出明确的红绿灯评级，并用**醒目的 Emoji** 开头：

*   🟢 **【可行】 (Feasible)**：功能完全合理，在现有技术栈内游刃有余。
    *   *行动*：无需赘述，直接进入 `sdd:plan` 或常规开发流程。
*   🟡 **【高风险 / 勉强可行】 (High Risk)**：能做，但代码会很丑陋、很脆弱（Hack），或者性能极差，维护成本飙升。
    *   *行动*：必须向用户明确指出"为什么这很危险"，并强烈建议使用折中方案。
*   🔴 **【不可行】 (Impossible)**：触碰了物理盲区或安全沙箱，或者当前依赖库的功能根本不支持。
    *   *行动*：**立刻拒绝！严禁提供任何试错代码！** 必须用通俗易懂的语言解释为什么做不到。

### 阶段 3：提出 B 计划 (折中/替代方案)
如果评级为 🟡 或 🔴，你**必须**提供至少 1-2 个能达到**类似业务目的**，但在技术上完全可行的替代方案。

**替代方案格式要求：**
- **替代方案 A：[方案简述]**
  - *为什么可行*：说明避开了哪些技术坑。
  - *妥协了什么*：说明用户体验或功能上和原需求比差在哪里。

## 触发此技能的响应模板示例

如果用户要求："让网页在后台静默录音并自动上传"（此需求涉及浏览器录音权限和后台存活限制）：

> 🔴 **【不可行】静默录音**
> 
> **原因**：现代浏览器出于隐私安全（安全沙箱机制），**绝对禁止**网页在没有用户明确授权和界面可见交互的情况下在后台静默调用麦克风。这是一个无法逾越的底线，任何代码层面的尝试都会被浏览器直接拦截并报错。
> 
> **替代方案 A：显式录音按钮 + 保持页面活跃**
> - **做法**：在页面上放置一个显眼的"开始录音"按钮，用户点击后请求权限。
> - **妥协了什么**：无法做到"无感静默"，必须有用户的初始点击动作。
> 
> 请问您接受这个替代方案吗？如果接受，我将开始为您规划这个按钮的实现。
