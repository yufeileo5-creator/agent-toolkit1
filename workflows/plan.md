# 计划模式工作流 (Plan Mode Workflow)

当用户提出任何涉及**代码修改、功能新增、Bug 修复、重构**的需求时，必须严格按照以下步骤执行。**绝对禁止在用户明确批准实施计划前执行任何代码修改或具有副作用的终端命令。** 所有输出和文档必须使用**中文**。

---

## 阶段一：规划 (PLANNING)

### 1. 需求澄清与上下文检索
- 仔细阅读需求。如有歧义或信息缺失，立即提问，绝不主观假设。
- **强制第一步：** 查阅现有 Knowledge Items (KIs) 及历史文档摘要，严禁在未检查历史决策的情况下从零开始分析。
- 检查根目录是否存在 `ARCHITECTURE.md`。若缺失，必须在回复中主动提议生成。
- 阅读现有 `ARCHITECTURE.md`、`README.md` 及相关设计文档，确保理解当前项目的架构分层和设计决策。

### 2. 方案设计与影响评估 (`implementation_plan.md`)
- 必须先分析并复用项目中已有的类似模式或工具函数，避免重新发明轮子。
- 在 artifact 目录下创建/更新 `implementation_plan.md`，强制包含以下结构：
  - **目标与意图**：要做什么、为什么做。
  - **架构合规声明**：确认新代码严格符合既有架构分层，未引入反向依赖（如 Domain 依赖 Infrastructure）。
  - **变更影响范围**：列出将修改的文件、间接受影响的导入/调用链，以及需同步更新的测试。
  - **详细修改清单**：按依赖顺序（被依赖方先改）罗列每个文件的具体修改计划。
  - **依赖与模块准入**：若需引入新第三方依赖或创建新顶层模块，必须在此单独列出说明原因并等待批准。
  - **风险与性能评估**：标注高频路径性能风险；若涉及删除/修改公共 API、改数据库、变环境变量，必须标记 `⚠️ BREAKING CHANGE`。

### 3. 任务分解拆解 (`task.md`)
- 在 artifact 目录下同步维护 `task.md`。
- 将实施计划拆解为细粒度的 `[ ]` 任务节点，确保每一个勾选都对应一个原子的、通过编译的代码状态。

### 4. 🚫 强制用户审批点 (BLOCKED ON USER)
- 使用 `notify_user` 提交 `implementation_plan.md` 供用户审核。
- **必须设置 `BlockedOnUser: true`**。
- **必须设置 `ShouldAutoProceed: false`**。
- 在用户回复诸如“通过”、“同意”、“去执行”前，**暂停一切工具调用**。若用户提出修改意见，更新文档并重新发起审批。

---

## 阶段二：执行 (EXECUTION)

### 5. 纪律性编码
获得批准后，按 `task.md` 的步骤严格执行。在编写任何代码时遵循：
- **完整性**：所有修改必须是完整可运行代码，严禁使用 `// ... existing code` 占位符。
- **类型与防御**：强制使用显式类型定义（如 TypeScript `interface`），禁用 `any`；所有外部输入必须校验。
- **模块化底线**：核心逻辑函数 $\le$ 40行，常规文件 $\le$ 300行。超出必须拆分。
- 每完成一步逻辑闭环，向系统同步当前进度与下一步意图。

---

## 阶段三：验证 (VERIFICATION)

### 6. 交付基线审查
- 代码修改完成后，必须**主动**在后台执行项目的验证命令（如 `npm run lint`, `tsc` 等）。
- 确认没有引发任何新的语法报错或类型检查失败。
- 若有测试套件，执行相关测试验证回归防护。

### 7. 闭环记录与总结 (`walkthrough.md`)
- 在 artifact 目录下创建/更新 `walkthrough.md`。
- 沉淀本次变更的总结、遇到的坑与根本原因分析（根因分析）、以及验证的实际结果。
- 对于重大架构决策，提议写入 `docs/` 下的 ADR 记录。
